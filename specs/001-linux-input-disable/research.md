# 调研记录: Linux 外置设备禁用内置输入

## 决策 1: 设备检测与分类方式

**Decision**: 使用 udev 事件监听检测输入设备的接入/移除，并根据设备属性区分内置与外置、键盘与指点设备。

**Rationale**: 事件驱动能满足低开销要求，同时 udev 提供稳定的设备属性用于分类，避免轮询与高频唤醒。

**Alternatives considered**:
- 轮询 `/dev/input` 设备列表: 简单但高开销，不符合宪章
- 依赖桌面环境或 X11/Wayland API: 与平台/会话绑定，兼容性差

## 决策 2: 禁用与恢复机制

**Decision**: 采用内核输入层独占抓取实现禁用，设备移除或异常时确保释放并恢复。

**Rationale**: 与桌面环境无关、覆盖面广，符合“最小权限与可恢复”原则。

**Alternatives considered**:
- 使用 `xinput` 或桌面环境开关: 仅适用于特定会话或桌面环境
- 停止内核驱动或卸载模块: 风险较高且可恢复性差

## 决策 3: 启用/禁用控制方式

**Decision**: 提供本地 CLI 作为控制接口（enable/disable/status），并将启用状态写入本地配置文件以持久化。

**Rationale**: CLI 便于用户操作与脚本集成，配置持久化可满足重启后保持状态。

**Alternatives considered**:
- 仅通过 systemd 启停服务: 粒度偏粗，不满足“功能开关”语义
- 仅使用配置文件手动修改: 易出错且不便于查询当前状态

## 决策 4: 按类型独立禁用与多设备计数

**Decision**: 维护外置键盘与外置鼠标的独立计数，只有对应类型计数为 0 时才恢复内置设备。

**Rationale**: 与需求“按类型独立、多个同类型外设需全部移除后恢复”一致，逻辑清晰。

**Alternatives considered**:
- 仅跟踪是否存在任意外设: 无法满足按类型独立的需求

## 决策 5: 低开销与性能验证

**Decision**: 仅在设备变更时执行逻辑，平稳期不主动轮询；性能预算以 CPU/内存/唤醒次数进行验证。

**Rationale**: 满足宪章对低开销与用户无干扰的要求，验证维度与规格一致。

**Alternatives considered**:
- 定时器轮询与状态比对: 增加唤醒与 CPU 开销

## 决策 6: 跨平台抽象边界

**Decision**: 核心策略逻辑与平台实现分离，Linux 仅提供设备发现与禁用实现，未来平台替换不改动策略。

**Rationale**: 直接对齐规格与宪章中的跨平台演进要求。

**Alternatives considered**:
- 将平台逻辑内嵌核心策略: 未来扩展成本高且易破坏既有行为

## 决策 7: 集成测试策略

**Decision**: 采用端到端集成测试覆盖“外设接入/移除 -> 内置设备禁用/恢复”关键路径，并在 CI 之外保留硬件实测流程。

**Rationale**: 真实设备组合复杂，端到端验证能显著降低回归风险。

**Alternatives considered**:
- 仅做单元测试: 无法覆盖设备插拔与状态迁移流程
- 仅依赖人工手测: 难以在合并门禁中保证稳定性
