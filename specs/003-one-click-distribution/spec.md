# 功能规格: 一键安装与自动化分发

**功能分支**: `003-one-click-distribution`
**创建日期**: 2026-03-10
**状态**: Draft
**输入**: 用户描述: "一键安装脚本与自动化发布。实现curl | bash的一键安装体验，通过GitHub Actions自动构建并发布预编译二进制文件到GitHub Releases，用户无需手动编译或配置构建环境。安装脚本自动检测系统架构，从Releases下载对应二进制并完成安装。"

> 说明: 本文档必须使用中文撰写，仅技术关键词、变量名、命令、路径等可保留英文。

## 用户场景与测试（必填）

### 用户故事 1 - 一键安装无需编译环境 (优先级: P1)

作为终端用户，我希望通过一条命令完成安装，无需配置Rust编译环境或下载源代码。

**为什么是此优先级**: 这是改善安装体验的核心目标，消除了当前最大的使用门槛。

**独立测试**: 在全新Linux系统上执行 `curl -sSL https://.../install.sh | bash`，无需预先安装任何构建工具即可完成安装。

**验收场景**:

1. **Given** 全新Linux系统无Rust环境, **When** 用户执行一键安装命令, **Then** 系统自动下载二进制并在1分钟内完成安装
2. **Given** 系统为x86_64架构, **When** 执行安装脚本, **Then** 自动检测并下载对应架构的二进制文件
3. **Given** 系统为aarch64架构, **When** 执行安装脚本, **Then** 自动检测并下载对应架构的二进制文件

---

### 用户故事 2 - 自动化构建与发布 (优先级: P1)

作为项目维护者，我希望在发布新版本时自动构建所有平台的二进制文件并发布到GitHub Releases。

**为什么是此优先级**: 这是实现一键安装的基础设施，没有自动发布就没有可下载的二进制文件。

**独立测试**: 推送Git tag后，GitHub Actions自动触发构建，生成多平台二进制文件并创建Release。

**验收场景**:

1. **Given** 代码仓库配置了GitHub Actions, **When** 推送版本标签（如v0.3.0）, **Then** 自动构建x86_64和aarch64架构的二进制文件
2. **Given** 构建成功完成, **When** 查看GitHub Releases, **Then** 找到对应版本的Release且包含所有架构的二进制文件
3. **Given** 用户访问Release页面, **When** 下载二进制文件, **Then** 文件完整且可执行

---

### 用户故事 3 - 安装脚本智能降级 (优先级: P2)

作为用户，我希望当GitHub Releases下载失败时，安装脚本能提供清晰的错误提示和替代方案。

**为什么是此优先级**: 保障安装体验的鲁棒性，避免用户在网络问题时卡住。

**独立测试**: 模拟GitHub访问失败场景，验证错误消息的清晰度和可操作性。

**验收场景**:

1. **Given** 无法访问GitHub Releases, **When** 执行安装脚本, **Then** 显示明确的错误信息和手动下载指引
2. **Given** 下载的二进制文件校验失败, **When** 执行安装脚本, **Then** 提示文件损坏并建议重新下载
3. **Given** 系统架构不受支持, **When** 执行安装脚本, **Then** 显示支持的架构列表并建议从源码编译

---

### 用户故事 4 - 版本检测与更新提示 (优先级: P3)

作为已安装用户，我希望运行现有版本时能检测到有新版本可用并获得升级指引。

**为什么是此优先级**: 提升用户体验，让用户及时获得bug修复和新功能。

**独立测试**: 安装旧版本后，检查版本检测功能是否正确识别新版本存在。

**验收场景**:

1. **Given** 已安装v0.2.0, **When** 运行 `aegis-input --version`, **Then** 显示当前版本并提示v0.3.0可用
2. **Given** 检测到新版本, **When** 显示升级提示, **Then** 包含升级命令或下载链接

---

### 边界情况

- GitHub API速率限制时如何避免安装失败？
- 下载过程中断后重新执行如何恢复或清理？
- 用户已经手动编译安装的情况下，一键脚本如何检测并处理？
- systemd服务已存在旧版本时如何平滑升级？

## 需求（必填）

### 功能性需求

- **FR-001**: 系统必须提供可通过 `curl | bash` 执行的一键安装脚本
- **FR-002**: 安装脚本必须自动检测系统架构（x86_64/aarch64）
- **FR-003**: 安装脚本必须从GitHub Releases下载对应架构的预编译二进制文件
- **FR-004**: 系统必须通过GitHub Actions在推送版本tag时自动构建多平台二进制文件
- **FR-005**: GitHub Actions必须自动将构建产物发布到GitHub Releases
- **FR-006**: 安装脚本必须验证下载文件的完整性（校验和或签名）
- **FR-007**: 安装失败时必须显示清晰的错误信息和可操作的解决建议
- **FR-008**: 安装脚本必须检测并提示不受支持的操作系统或架构
- **FR-009**: 系统必须在Release中包含版本校验信息（SHA256SUMS）
- **FR-010**: 安装脚本必须提供 `--version` 或 `--check` 选项来检测新版本

### 关键实体（若功能涉及数据）

- **Release元数据**: 包含版本号、发布日期、各架构二进制文件的下载链接和校验和
- **安装状态**: 记录当前安装版本、安装时间、安装来源（一键脚本或手动编译）
- **版本信息**: 本地版本号与远程最新版本号的对比结果

## 假设与依赖

- GitHub仓库可公开访问， Releases页面可供匿名下载
- 用户系统已连接互联网且可访问GitHub
- 用户具有sudo权限以完成系统级安装
- 初始支持的Linux发行版为Debian/Ubuntu/Mint系列（systemd）
- 二进制文件静态链接，不依赖特定系统库版本
- GitHub Actions有足够的构建配额

## 成功标准（必填）

### 可衡量结果

- **SC-001**: 全新用户从执行命令到完成安装的平均时间不超过60秒
- **SC-002**: 一键安装成功率（网络正常情况下）不低于95%
- **SC-003**: GitHub Actions构建和发布全流程在推送tag后10分钟内完成
- **SC-004**: 安装脚本支持的架构覆盖率达到已发布二进制文件的100%
- **SC-005**: 安装失败场景下，错误信息的可操作性评分（用户反馈）不低于4/5
- **SC-006**: 版本检测准确率达到100%（无误报或漏报）

### 技术约束（用于实现参考，非业务需求）

- 安装脚本必须为POSIX兼容的shell脚本（sh/bash）
- GitHub Actions工作流使用Rust交叉编译生成多架构二进制
- 二进制文件启用静态链接以提升兼容性
- Release中包含SHA256校验和文件供验证
