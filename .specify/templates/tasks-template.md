# 任务清单: [FEATURE NAME]

**输入**: 来自 `/specs/[###-feature-name]/` 的设计文档
**前置条件**: plan.md（必需）, spec.md（必需，含用户故事）, research.md, data-model.md, contracts/

**测试**: 集成测试为必须项。每个用户故事至少包含 1 条端到端集成测试。单元测试可选。

**组织方式**: 任务按用户故事分组，确保每条故事可独立实现与测试。

## 格式: `[ID] [P?] [Story] 描述`

- **[P]**: 可并行执行（不同文件、无依赖）
- **[Story]**: 所属用户故事（例如: US1, US2, US3）
- 描述中必须包含明确的文件路径

## 路径约定

- **单项目**: `src/`, `tests/` 位于仓库根目录
- **Web 应用**: `backend/src/`, `frontend/src/`
- **移动端**: `api/src/`, `ios/src/` 或 `android/src/`
- 下方路径假设单项目结构，请按 plan.md 调整

## 依赖与执行顺序

### 阶段依赖

- **初始化（Phase 1）**: 无依赖，可立即开始
- **基础能力（Phase 2）**: 依赖 Phase 1 完成，阻塞所有用户故事
- **用户故事（Phase 3+）**: 依赖 Phase 2 完成
  - 完成后可并行（如有人力）
  - 或按优先级顺序执行（P1 → P2 → P3）
- **打磨（最终阶段）**: 依赖所有选定用户故事完成

### 用户故事依赖

- **用户故事 1 (P1)**: Phase 2 完成后可开始，无依赖
- **用户故事 2 (P2)**: Phase 2 完成后可开始，可能集成 US1
- **用户故事 3 (P3)**: Phase 2 完成后可开始，可能集成 US1/US2

### 每条用户故事内部顺序

- 集成测试必须先写并确保失败
- 模型先于服务
- 服务先于接口/端点
- 核心实现先于集成
- 完成后必须独立验证

### 并行机会

- Phase 1 中标记 [P] 的任务可并行
- Phase 2 中标记 [P] 的任务可并行
- Phase 2 完成后，各用户故事可并行
- 同一用户故事中标记 [P] 的测试与模型可并行
- 不同用户故事可由不同成员并行

## 实施策略

### MVP 优先（仅用户故事 1）

1. 完成 Phase 1: 初始化
2. 完成 Phase 2: 基础能力（关键）
3. 完成 Phase 3: 用户故事 1
4. **停止并验证**: 用户故事 1 端到端集成测试必须通过
5. 视情况部署/演示

### 递增交付

1. 完成初始化 + 基础能力 -> 基础就绪
2. 添加用户故事 1 -> 独立测试 -> 部署/演示（MVP）
3. 添加用户故事 2 -> 独立测试 -> 部署/演示
4. 添加用户故事 3 -> 独立测试 -> 部署/演示
5. 每条故事都应在不破坏前序功能的前提下增加价值

### 并行团队策略

多人协作时:

1. 团队完成初始化 + 基础能力
2. 基础完成后:
   - 开发者 A: 用户故事 1
   - 开发者 B: 用户故事 2
   - 开发者 C: 用户故事 3
3. 各故事独立完成并集成

## 备注

- [P] 任务 = 不同文件、无依赖
- [Story] 标记用于任务与用户故事的追踪
- 每条用户故事必须可独立完成与测试
- 集成测试先行并确保失败
- 在每个检查点可停止并验证
- 避免: 含糊任务、同文件冲突、跨故事依赖导致不可独立测试
